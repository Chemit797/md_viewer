## 第一类：序列/表位预测工具

---

### NetMHCIIpan-4.3

**原始功能（它本来是干嘛的）**
```
端对端：
    输入：任意肽序列（9-25aa）+ HLA-II等位基因
    输出：结合亲和力IC50（nM）/ 呈递概率百分位
    
任务：预测这段肽能不能装上MHC-II分子展示出来
训练数据：IEDB实测结合亲和力 + 质谱洗脱的天然MHC-II配体
覆盖：116个MHC-II分子，400万数据点
成熟度：高，ADA链路[4]的标准工具
```

**我们怎么用**
```
位置：Pipeline第一步，序列侧特征提取

操作流程：
    1. 把治疗性多肽（如exendin-4，39aa）滑窗切成15aa片段
    2. 每个片段 × 6-8个代表性HLA-DR等位基因
    3. 调NetMHCIIpan API跑预测
    4. 输出：每个窗口的IC50矩阵
    5. 聚合成单一"呈递强度分数"

我们取的结果：呈递强度分数 → 作为激活信号输入模型2
```

**迁移时会遇到的问题**
```
问题1：滑窗聚合方式无标准
    多个窗口各有IC50，怎么合成一个分数没有定论
    选最低IC50 vs 平均 vs 高亲和力窗口数量
    不同聚合方式结果差异可能很大

问题2：修饰多肽无法编码
    semaglutide含Aib（非天然氨基酸）+ 脂肪酸链
    NetMHCIIpan只接受标准20种氨基酸
    修饰信息完全丢失

问题3：HLA等位基因选哪些
    工具支持几百种HLA-DR/DP/DQ
    默认参数是欧美人群频率
    如果针对中国患者，等位基因频率需要重新校准

问题4：训练分布偏移
    训练数据是天然蛋白表位
    我们的输入是人工设计的治疗性多肽
    在分布外的序列上预测可靠性未知
```

**解决思路**
```
问题1：做消融实验
    把几种聚合方式都跑一遍
    用已知免疫原性高/低的多肽验证哪种最准
    选表现最好的方式，在论文里说明

问题2：忽略修饰，用骨架序列
    PMID 29758051已有先例（exendin-4骨架去掉修饰跑NetMHCIIpan）
    论文里写明limitation：对重度修饰多肽预测可靠性有限
    未来工作：开发能接受修饰氨基酸的编码方法

问题3：换用中国人群HLA频率数据
    参考中国骨髓库（CMDP）的HLA频率统计
    选覆盖中国90%人群的代表性等位基因组合

问题4：接受并量化
    这是所有迁移学习都有的问题
    做域适应实验：看预测值分布是否系统性偏移
    在论文中诚实报告
```

### TLimmuno2

**原始功能（它本来是干嘛的）**
```
端对端：
    输入：13-21aa肽序列 + HLA-II等位基因
    输出：T细胞激活概率（0-1，分类）

任务：预测这段肽装上MHC-II之后，CD4+ T细胞会不会被激活
训练数据：IEDB中病原体/肿瘤来源的T细胞激活实验数据
架构：LSTM + 迁移学习（从BA数据预训练）
真实性能：独立测试集AUC ≈ 0.74（交叉验证0.87被高估了）
覆盖：ADA链路[5]-[6]
```

**我们怎么用**
```
位置：Pipeline可选步骤，在NetMHCIIpan之后做二次过滤

操作流程：
    1. NetMHCIIpan筛出高结合亲和力窗口（IC50 < 500nM）
    2. 把这些短窗口送进TLimmuno2
    3. 进一步判断：结合了之后T细胞真的会激活吗
    4. 输出：激活概率，用来补充调整激活强度分数

注意：这步是可选的，不是必须的
    数据少的情况下可以先跳过
    直接用NetMHCIIpan分数进入下一步
```

**迁移时会遇到的问题**
```
问题1：长度硬限制
    TLimmuno2只接受13-21aa
    多数治疗性多肽远超这个范围（exendin-4是39aa）
    必须先切割，但切割破坏了整条肽的上下文信息

问题2：训练分布严重偏移
    训练数据：病原体/肿瘤新抗原
    我们的输入：人工设计治疗性多肽
    这两类序列的氨基酸组成和物理化学性质差异很大

问题3：性能基准不可靠
    AUC 0.74是在病原体/肿瘤表位上测的
    在治疗性多肽上的真实性能完全未知
    可能更低
```

**解决思路**
```
问题1：滑窗后送入
    先用NetMHCIIpan确认高亲和力窗口
    只把这些短窗口送TLimmuno2
    聚合时用窗口级别的激活概率加权
    这是现有最合理的workaround，有文献先例

问题2和3：降低权重或不用
    如果发现TLimmuno2在治疗性多肽上表现不稳定
    可以从"核心步骤"降级为"参考步骤"
    论文中做消融实验：有TLimmuno2 vs 没有TLimmuno2
    如果加了没有提升，就不放在最终模型里
```

### EpiMatrix / EpiVax ISPRI

**原始功能**
```
端对端：
    输入：蛋白/多肽序列
    输出：T细胞表位密度分数 / ADA风险评级

任务：评估治疗性蛋白药物的免疫原性风险
训练：以蛋白药物（主要是单抗）为主
成熟度：商业声明87.5%准确率（24种单抗，EpiVax白皮书）
        无同行评议验证数据，是黑盒商业工具
```

**我们怎么用**
```
不直接用于建模
主要用途：作为比较基线

在论文中：
    用EpiMatrix对同一批多肽做预测
    和我们的方法比较ADA风险预测性能
    如果我们比它强 → 体现创新价值
    如果差不多 → 至少说明我们的机理解释性更好
```

**迁移时会遇到的问题**
```
问题1：商业黑盒，不开源
    无法知道内部逻辑，无法复现
    无法做消融实验理解它为什么做出某个预测

问题2：单抗验证数据不能外推到多肽
    声称的87.5%准确率来自24种单抗
    多肽的验证数据为零

问题3：不提供代谢机理特征
    纯序列驱动，没有T细胞侧的任何信息
    这是我们的差异化所在
```

**解决思路**
```
不解决——这是我们的竞争对手，不是我们要用的工具
把它作为baseline对比就够了
我们的优势是机理可解释性，不需要打败它的每一个指标
```

### IEDB分析工具（在线预测资源）

**原始功能**
```
端对端：
    输入：肽序列 + HLA
    输出：MHC结合预测 / T细胞表位分析

本质：是NetMHCIIpan等工具的在线界面
      同时提供IEDB数据库的查询功能
成熟度：高，是标准学术工具
```

**我们怎么用**
```
工具本身：可以用，等价于NetMHCIIpan
数据库：治疗性多肽T细胞数据为零，不可用

具体用途：
    验证阶段：把我们预测高风险的多肽
              拿到IEDB查有没有已发表的T细胞实验记录
              作为外部验证的参考
```

**迁移时的问题和解决思路**
```
问题：数据库里没有我们需要的多肽T细胞数据
解决：不用IEDB数据训练模型，数据来源转向复旦合作
      IEDB工具本身可以正常调用
```

## 第二类：GEM / 代谢模型工具

---

### CD4T1670（Han 2016）

**原始功能**
```
端对端：
    输入：代谢底物浓度约束（葡萄糖、氨基酸等）
    输出：各代谢反应通量分布（FBA求解结果）

任务：模拟naive CD4+ T细胞的代谢网络
规模：1670个代谢反应（注意：是反应数，不是基因数）
PMID：26646474，Mol BioSyst 2016，第一作者Han Feifei
功能：可以模拟药物干预对代谢通量的影响
```

**我们怎么用**
```
用途：参照和起点，不是直接用

1. 参照naive T细胞代谢基线
   T细胞激活前的代谢状态来自这个模型
   帮助我们设定GEM约束的"静息态"边界值

2. 学习方法论
   它的FBA设置方式 → 直接借用到Tfh GEM的构建
   它验证代谢网络的方法 → 复用

注意：它只有naive态，没有激活态和Tfh分化态
     不能直接用来预测Tfh代谢
```

**迁移问题和解决思路**
```
问题：只有naive，我们需要激活态和Tfh分化态
解决：以它为参照，用tINIT+scRNA-seq数据
     构建Tfh特异性GEM（见下方"我们自建的模型1"）
     CD4T1670提供方法论模板，不提供最终模型
```

### Sen 2021 GEM（Th1/Th2/Th17/iTreg）

**原始功能**
```
端对端：
    输入：各亚型T细胞的基因表达数据
    输出：亚型特异性代谢通量差异

任务：量化Th1/Th2/Th17/iTreg四种亚型的代谢网络差异
PMID：34758307，Cell Reports 2021，第一作者Partho Sen
验证：脂质组学 + siRNA敲低实验（SPT敲降影响Th17功能）
关键发现：glycosphingolipid通路在Th17分化中的决定性作用
模型已存入BioModels（MODEL2101270002）可下载
```

**我们怎么用**
```
用途：最直接的方法论模板

1. 直接借用构建流程
   他们从Human-GEM裁剪出亚型GEM的步骤
   → 我们用同样的步骤裁剪出Tfh GEM

2. 作为比较参照
   论文中：我们的Tfh GEM vs Sen 2021的iTreg GEM
   展示Tfh和Treg在代谢上的异同

3. 下载BioModels里的模型文件
   作为建Tfh GEM的起点格式参考
```

**迁移问题和解决思路**
```
问题1：没有Tfh
    这是整个项目存在的原因，不是障碍是机会

问题2：他们的实验验证方法（脂质组学+siRNA）我们未必能做
    如果没有实验室条件
    可以用已发表的Tfh代谢文献做计算层面的验证
    （Mayberry 2022综述提供了Tfh代谢特征的参照）

解决：直接用他们的方法论建Tfh GEM
     验证策略换成计算验证+文献对比
     如果有复旦合作，可以追加体外代谢实验验证
```

端对端：

### Human-GEM / Recon3D

**原始功能**
```
端对端：
    这不是一个"预测工具"，是一个知识库

内容：全人类代谢反应的完整网络
     几千个反应、代谢物、基因-反应对应关系
用途：作为构建context-specific GEM的基础
     所有人类细胞类型的GEM都从这里裁剪

Human-GEM：开源，GitHub维护，推荐使用
Recon3D：更老的版本，覆盖略有差异
```

**我们怎么用**
```
这是Tfh GEM的原材料

流程：
    Human-GEM（全集）
         ↓ 输入Tfh scRNA-seq表达数据
    tINIT算法裁剪
         ↓ 删掉Tfh里不活跃的反应
    Tfh特异性子网络
         ↓ 就是我们的模型1

不直接用Human-GEM跑预测
它太大，包含所有细胞类型的反应
必须裁剪成Tfh特异版本
```

**迁移问题和解决思路**
```
问题1：Human-GEM的免疫细胞部分注释不完整
    Human1文档说免疫细胞相关子系统标注较弱
    某些Tfh特异性代谢反应可能缺失或标注错误

问题2：从全集裁剪后可能出现孤立反应（gap）
    某些反应的底物没有生成来源，形成代谢孤岛
    FBA会出现infeasible解

解决思路：
    问题1：用Sen 2021和文献补充标注
           特别是FAO和谷氨酰胺代谢相关反应
    问题2：gap-filling算法（COBRA Toolbox里有）
           自动填补孤立反应，然后手动核查
```

### tINIT / iMAT / GIMME

**原始功能**
```
这三个都是"context-specific GEM构建算法"
功能相同，实现方式略有差异

端对端（以tINIT为例）：
    输入：
        ① 基础GEM（Human-GEM，全集）
        ② 目标细胞类型的基因表达数据（RNA-seq）
    输出：
        该细胞类型的特异性代谢子网络

原理：
    基因表达量高 → 对应代谢反应"开着"→ 保留
    基因表达量低 → 对应代谢反应"关着"→ 删除
    同时保证剩余网络在数学上连通且可解
```

**我们怎么用**
```
这是构建Tfh GEM的核心操作步骤

具体操作：
    1. 下载E-MTAB-13687（HCA扁桃体细胞图谱）
       提取其中GC-Tfh细胞群的平均基因表达谱
    2. 把表达谱 + Human-GEM喂给tINIT
    3. tINIT输出Tfh特异性代谢子网络
    4. 这就是我们的模型1（Tfh GEM）

工具获取：
    COBRA Toolbox（MATLAB）有完整实现
    也有Python版本（cobrapy）
    tINIT算法来自Agren et al. 2012，有现成代码
```

**迁移问题和解决思路**
```
问题1：需要高质量的Tfh scRNA-seq数据
    如果E-MTAB-13687里的Tfh细胞FAO基因覆盖不够
    算法就会把FAO通路错误地关掉
    → 这是第0阶段必须先核查的

问题2：算法参数敏感
    tINIT里的阈值设置（多低的表达量算"关着"）
    不同参数会产生差异显著的子网络
    
问题3：结果难以唯一验证
    不同参数产生的Tfh GEM哪个"更对"
    需要和已知Tfh代谢特征比对

解决思路：
    问题1：先下载数据，检查CPT1A、HADHA等FAO基因表达量
           如果覆盖不足，补充bulk RNA-seq数据或联系复旦
    问题2：做参数敏感性分析，报告结果稳健性
    问题3：用Mayberry 2022综述的Tfh代谢特征作为验证标准
           （低GLUT1/HK2，高CPT1A，低HIF-1α靶基因）
```

### DLKcat / UniKP

**原始功能**
```
端对端：
    输入：酶的氨基酸序列 + 底物结构
    输出：kcat值（酶的催化速率常数）

任务：预测酶促反应速率，用于构建酶约束GEM（ecGEM）
ecGEM比普通FBA多了"酶的量是有限的"这个约束
使得代谢通量预测更接近真实细胞
```

**我们怎么用**
```
MVP阶段不用。

只有在以下情况才考虑：
    普通FBA的Tfh GEM验证结果不够准确
    需要加入酶量约束来提高精度
    → 这是第二版或第三版论文的事

第一版先用普通FBA
够用就不加复杂度
```

## 第三类：虚拟细胞 / 基础模型

---

### scGPT / Geneformer

**原始功能**
```
端对端（以scGPT为例）：
    输入：单细胞基因表达向量
    输出：细胞状态嵌入（高维向量）

任务：学习单细胞基因表达的通用表示
      支持：细胞类型注释、扰动预测、基因调控网络推断
架构：Transformer，在大量scRNA-seq数据上预训练
特点：理解细胞整体状态，但不输出代谢通量
```

**我们怎么用**
```
可选增强，不是核心

用法1：辅助Tfh GEM构建
    用scGPT从E-MTAB-13687提取Tfh细胞状态嵌入
    作为tINIT算法的补充约束
    （除了基因表达量，还加入细胞状态信息）

用法2：论文扩展
    第一版不用，第二版可以加进来
    展示"GEM + 虚拟细胞基础模型"的结合
    对齐项目最初的"GEM + 虚拟细胞"框架
```

**迁移问题和解决思路**
```
问题1：不输出代谢通量
    scGPT的输出是基因表达空间的嵌入
    不能直接作为FBA的约束条件
    需要一个额外的映射层

问题2：和GEM的接口没有现成方案
    把scGPT嵌入转化成代谢约束
    目前没有发表过的方法

解决思路：
    问题1：把scGPT嵌入作为ML分类器（模型3）的额外输入特征
           而不是作为GEM的约束
           这样绕开了接口问题
    问题2：如果要做深度整合，这本身就是一个创新点
           但复杂度高，放到第二版
```

## 第四类：数据集

---

### IEDB数据库

**原始内容**
```
内容：T细胞表位、B细胞表位、MHC配体的实验数据
规模：~51万条T细胞实验，~140万条B细胞实验
主体：病原体（流感、HIV、COVID等）/过敏原/自身抗原/移植抗原
治疗性多肽数据：为零（已核实）
```

**我们怎么用**
```
MHC-II结合数据：可以用
    下载IEDB里的MHC-II结合亲和力数据集
    用于理解NetMHCIIpan的训练分布
    也可以用来做预训练（迁移学习的第一步）

T细胞激活数据：不可用于治疗性多肽
    治疗性多肽数据为零
    只能用病原体数据做迁移学习的起点
    
外部验证：可以查
    对已知免疫原性结果的多肽
    去IEDB看有没有相关T细胞实验记录
    作为额外验证点
```

**问题和解决思路**
```
核心问题：治疗性多肽T细胞数据为零
唯一解决方案：
    → 迁移学习（用单抗数据迁移到多肽）
    → 复旦合作获取实验数据
    → 这两条路都需要追加工作，不是工具能解决的
```

## 第五类：MAPPs数据集

### Barra 2020 / Rooney 2020 MAPPs数据

**原始内容**
```
MAPPs（MHC-Associated Peptide Proteomics）：
    用质谱实测细胞MHC-II上实际装载的肽
    是最直接的MHC-II呈递实验数据

Barra 2020（PMID 32655572）：
    对infliximab/rituximab等单抗做MAPPs
    实测呈递肽段 → 训练MHC-II呈递预测模型
    Spearman ρ 0.42-0.64

Rooney 2020：
    公开的MAPPs数据集（MassIVE MSV000085375）
    同样是单抗来源
```

**我们怎么用**
```
数据本身：不直接用
    数据对象是单抗，不是多肽

方法论参照：
    如果未来需要做体外MAPPs实验验证
    Barra 2020提供了完整的实验设计模板：
    如何处理样本 → 如何做质谱 → 如何分析数据

迁移学习起点：
    单抗MAPPs数据 → 训练MHC-II呈递预测
    再迁移到多肽
    这是数据最少时的保底策略
```

**问题和解决思路**
```
问题：单抗 → 多肽的迁移，蛋白质和多肽的加工方式不同
解决：在迁移时加入"肽长"作为特征
     区分单抗来源的长片段和多肽药物本身
     明确在论文中报告迁移效果
```

## 总览：工具在Pipeline里的位置

```
治疗性多肽序列
    ↓
[NetMHCIIpan]        → 序列侧，呈递概率        → 借用，调API
    ↓
[TLimmuno2]          → 激活过滤（可选）         → 借用，调API
    ↓
[模型2: 映射函数]     → IC50 → 代谢约束         → 自建，无先例
    ↓
[模型1: Tfh GEM]     → FBA求解代谢通量          → 自建
[Human-GEM]          → 原材料                   → 下载使用
[tINIT]              → 裁剪工具                 → 运行算法
[E-MTAB-13687]       → 输入数据                 → 下载分析
    ↓
[模型3: ML分类器]    → 通量→分化预测            → 自建
[复旦实验数据]       → 训练标签                 → 合作获取
    ↓
[输出] Tfh倾向分数 / ADA风险预测

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
参照/对比（不在主pipeline里）：
    [EpiMatrix]       → 比较基线
    [Sen 2021 GEM]    → 方法论模板
    [CD4T1670]        → 静息态参照
    [scGPT]           → 可选增强（第二版）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
数据来源：
    [IEDB MHC-II结合数据] → NetMHCIIpan预训练理解
    [复旦PBMC数据]        → 模型3训练标签（关键）
    [单抗MAPPs数据]       → 迁移学习保底选项
```
